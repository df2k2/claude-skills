# Tailwind CSS v4 in Hyvä

## When to read this

Hyvä 1.4.x and newer ships with Tailwind v4. You're configuring tokens, debugging missing styles, customizing colors, building CSS, or trying to add a Tailwind plugin. (For older Hyvä on Tailwind v3, see `upgrading.md`.)

## What changed in v4 (vs. v3)

- **CSS-first config.** No `tailwind.config.js`. Theme values are defined in CSS via `@theme { … }` blocks.
- **Oxide engine** (Rust) — much faster builds.
- **Automatic content detection** — Tailwind v4 scans automatically, but Hyvä uses its own scanner that follows the `hyva.config.json` includes (because templates live across many directories in Magento).
- **Renamed/removed utilities.** `bg-opacity-*` → slash syntax (`bg-black/50`). `flex-grow` → `grow`. `overflow-ellipsis` → `text-ellipsis`. `decoration-slice/clone` → `box-decoration-slice/clone`.
- **CSS variables exposed.** Every theme value becomes a `--*` custom property.

## File structure (Hyvä 1.4.x)

```
web/tailwind/
├── hyva.config.json           # JSON config: paths, plugins, simple tokens
├── tailwind-source.css        # Top-level CSS that imports everything
├── package.json
├── package-lock.json
├── base/                      # Hyvä preflight + classless defaults
├── components/                # toolbar, button, form, messages, etc.
├── generated/                 # auto-generated by hyva-sources / hyva-tokens (don't edit)
│   ├── hyva-source.css
│   └── hyva-tokens.css
├── theme/                     # page-specific or custom styles
└── utilities/                 # custom utility classes
```

## `hyva.config.json`

The top-level configuration. Common keys:

```json
{
  "tailwind": {
    "include": [
      { "src": "vendor/hyva-themes/magento2-default-theme" },
      { "src": "app/code/Acme/Module" }
    ]
  },
  "plugins": [
    "@tailwindcss/typography"
  ],
  "tokens": {
    "colors": {
      "primary": {
        "lighter": "#93c5fd",
        "DEFAULT": "#1e40af",
        "darker":  "#1e3a8a"
      }
    }
  }
}
```

- `tailwind.include[].src` — paths the build scans for class names. Always include the parent default theme; add `app/code/<Vendor>/<Module>` if a third-party module's templates use Tailwind classes you want shipped.
- `plugins` — npm package names (must also be in `package.json`). `@tailwindcss/typography`, `@tailwindcss/forms`, etc.
- `tokens` — inline design tokens. Generates `generated/hyva-tokens.css`.

## `tailwind-source.css`

The entry point Tailwind compiles. Default Hyvä:

```css
@import "tailwindcss";
@import "./generated/hyva-source.css";
@import "./generated/hyva-tokens.css";

@import "./base/index.css";
@import "./components/index.css";
@import "./utilities/index.css";
@import "./theme/index.css";
```

If you've migrated from v3 and converted a `tailwind.config.js`, append:
```css
@import "./generated/tailwind.config.css";
```

## Defining theme values in CSS

Use `@theme` blocks anywhere in your CSS (often `tailwind-source.css` or `theme/index.css`):

```css
@theme {
    --color-primary: oklch(0.45 0.18 260);
    --color-primary-darker: oklch(0.35 0.18 260);
    --color-secondary: #ec4899;

    --font-display: "Inter", sans-serif;

    --spacing-page: 1280px;

    --breakpoint-3xl: 1920px;
}
```

Each `--color-X` becomes a class: `bg-X`, `text-X`, `border-X`, etc. `--font-X` becomes `font-X`. `--spacing-X` becomes `p-X`, `m-X`, `gap-X`. See [tailwindcss.com/docs/theme](https://tailwindcss.com/docs/theme) for the full list of recognized prefixes.

## Design tokens (DTCG / Tokens Studio)

For larger projects, define tokens in a separate file (DTCG format) and use the `hyva-tokens` command:

```bash
npx hyva-tokens
```

This reads `hyva.config.json`'s `tokens` block (or a separate file) and writes `generated/hyva-tokens.css`. Run after every token change.

DTCG format example:
```json
{
    "color": {
        "primary": {
            "$value": "#1e40af",
            "$type": "color"
        }
    }
}
```

## CSS components and utilities

Custom components in `components/`:
```css
/* components/button.css */
.btn {
    @apply inline-flex items-center justify-center px-4 py-2 rounded-md font-medium transition;
}
.btn-primary {
    @apply bg-primary text-white hover:bg-primary-darker;
}
```

Custom utilities in `utilities/`:
```css
/* utilities/scrollbar.css */
@utility scrollbar-thin {
    scrollbar-width: thin;
}
```

Note the new `@utility` syntax for v4 (replaces `@layer utilities` in v3 for custom utility definitions).

## Building

From the child theme's `web/tailwind/`:

```bash
npm ci                   # one-time install
npm run build            # production build → web/css/styles.css
npm run watch            # rebuild on change (alias since 1.4.0: npm start)
npm run browser-sync -- --proxy http://your-magento.test  # live reload
```

The output `styles.css` is then served by Magento's static content deploy. In production:
```bash
bin/magento setup:static-content:deploy -f
```

Removed in 1.4.0: `npm run build-prod`, `npm run build-dev` — both folded into `npm run build` and `npm run watch`.

## Dynamic class names — purging gotcha

Tailwind only includes classes it sees as **literal strings** in scanned files. So this won't work:

```php
<div class="bg-<?= $escaper->escapeHtmlAttr($color) ?>-500">…</div>
```

If `$color` is `red`, Tailwind doesn't know to compile `bg-red-500` because it's never written literally.

Fixes (in order of preference):

### 1. Whitelist via the safelist
Mention the classes literally somewhere in a scanned file:
```html
<!-- comment in a scanned phtml -->
<!-- safelist: bg-red-500 bg-green-500 bg-blue-500 -->
```
Or in a CSS comment in `tailwind-source.css`:
```css
/* safelist: bg-red-500 bg-green-500 bg-blue-500 */
```

### 2. Map dynamic values to literal class names

```php
<?php
$colorClasses = [
    'red'   => 'bg-red-500',
    'green' => 'bg-green-500',
    'blue'  => 'bg-blue-500',
];
$class = $colorClasses[$color] ?? 'bg-gray-500';
?>
<div class="<?= $escaper->escapeHtmlAttr($class) ?>">…</div>
```

### 3. Use arbitrary values
For one-off cases:
```html
<div class="bg-[#ff0000]">…</div>
```
Tailwind's JIT generates this on demand, even if it appears literally in templates.

### 4. CSS variables in inline `style`
For runtime dynamic values:
```html
<div :style="`--accent: ${productAccent}`" class="bg-[var(--accent)]">…</div>
```

## Including more files in the scan

Add to `hyva.config.json`:
```json
{
  "tailwind": {
    "include": [
      { "src": "vendor/hyva-themes/magento2-default-theme" },
      { "src": "app/code/Acme/Module" },
      { "src": "vendor/some-vendor/some-module/src" }
    ]
  }
}
```

The build scans `*.phtml`, `*.xml`, and `*.html` by default. To scan other extensions, you can add a `pattern` per source.

Rebuild after changing `hyva.config.json`: `npm run build`.

## Plugins (e.g., Typography, Forms)

```bash
cd web/tailwind
npm install --save-dev @tailwindcss/typography @tailwindcss/forms
```

Add to `hyva.config.json`:
```json
{
  "plugins": [
    "@tailwindcss/typography",
    "@tailwindcss/forms"
  ]
}
```

Rebuild.

## Dark mode

Tailwind v4 dark mode is configured via CSS:

```css
@theme {
    --color-bg: white;
    --color-text: black;
}

@theme dark {
    --color-bg: #111;
    --color-text: #eee;
}
```

Or use the `dark:` variant directly: `bg-white dark:bg-gray-900`. Hyvä's preflight respects `prefers-color-scheme` if you opt in.

## Email styles

Hyvä compiles a separate stylesheet for emails (because email clients don't run JS). See `web/tailwind/email/` (or `email-tailwind/`) for the email build pipeline. Same Tailwind v4, but inlining of styles happens at email render time.

## Troubleshooting

**Style isn't applied:**
1. Did you rebuild? `npm run build` from `web/tailwind/`.
2. Did Magento clear its cache? `bin/magento cache:flush`.
3. In production, did you redeploy static content? `bin/magento setup:static-content:deploy -f`.
4. Is the class in a file that's actually scanned? Check `hyva.config.json` includes.
5. Is the class dynamic (built from a variable)? See "Dynamic class names" above.
6. Is there a stale `web/css/styles.css` from before? Delete it and rebuild.

**Build fails on Node version:**
Hyvä 1.4.x requires Node.js 20+. Check `node --version`.

**Old class names from v3 still in templates:**
Use the upgrade helper:
```bash
./vendor/bin/find-deprecated-classes.js app/design/frontend/Acme/default/
```
This produces a report of `bg-opacity-*`, `flex-grow-*`, etc. that need converting.

## Migrating from Tailwind v3 (Hyvä 1.3.x → 1.4.x)

Use the wrapper script:
```bash
./vendor/bin/update-to-tailwind-v4.js app/design/frontend/Acme/default/
```

This:
1. Backs up `web/tailwind/` to `web/tailwind.backup.<date>`
2. Copies the v4 source files from the new default theme into your child
3. Converts your `tailwind.config.js` to `generated/tailwind.config.css`
4. Generates a `tailwind-deprecated-report.md` for class renames

Then manually:
- Re-add any custom theme values from your old config to `@theme` blocks
- Apply the deprecation report (or use the AI prompt in the upgrade helper to automate it)
- Rebuild and test

Detailed steps: see the Hyvä upgrade docs and `upgrading.md` reference.
